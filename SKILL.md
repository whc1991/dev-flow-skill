# 开发流程

## 核心原则

将本技能的调用视为当前开发单元的编码授权，但不视为对当前 Git 分支的确认。默认用户已完成需求梳理并明确了当前功能点或 PR；确认分支并核对范围后直接开发，不重复规划整个需求，也不设置范围的二次审批。

始终由用户决定开发单元、串行顺序和推进节奏。一次只完成一个用户指定的开发单元，完成后只告知下一待开发单元并停止，等待用户确认，不提前开发后续单元。

使用简短调用格式：`$dev-flow <开发单元>`，例如 `$dev-flow PR4`、`$dev-flow 登录超时修复`。将技能名后的内容直接视为当前开发单元标识，不要求用户补写“使用”“开发”或版本号等重复信息。

## 开始开发

1. 读取适用的仓库规则，并按“确认开发分支”完成分支安全检查。分支确认前只执行只读检查，不修改任何项目文件。
2. 分支确认后，以技能名后的内容为开发单元标识，读取当前仓库的需求记录、PR 描述、相关代码和测试，并检查当前工作区状态。
3. 优先使用当前仓库和当前对话上下文定位开发单元。标识能够唯一定位时，不要求用户补充版本号或完整标题；找不到对应范围或存在多个同名候选时，再请求用户澄清。
4. 根据现有证据简短说明当前单元的目标、范围和明确的非目标，以此完成范围核对。
5. 范围清楚时立即编码，不要求用户再次回复“开始开发”。
6. 仅在范围缺失、信息互相冲突，或必须由用户作出重要产品或数据决策时暂停并提问。
7. 普通技术细节应遵循项目现有模式自行判断，不反复请求用户决定。

## 确认开发分支

1. 任何项目文件修改前，使用只读 Git 命令确认当前仓库、当前分支和工作区状态。
2. 将“用户明确确认”作为开始编码的必要条件。满足以下任一情况才视为分支已经确认：
   - 用户在当前请求或当前对话上下文中明确指定了目标分支，且当前分支与其完全一致；
   - 模型报告当前分支名称后，用户明确回复该分支正确或同意在该分支开发。
3. 不根据版本号、PR 名称、分支命名习惯、远程默认分支、历史对话中的其他任务或模型推测，擅自认定当前分支正确。
4. 用户尚未指定或确认目标分支时，报告当前分支名称、工作区是否存在未提交修改，并请求确认。得到确认前停止编码，但可以继续进行不会修改文件的需求和代码检查。
5. 当前分支与用户指定分支不一致时，明确报告两者并停止。不得自行创建、切换、重命名、合并、变基或删除分支；等待用户处理或明确指令。
6. 处于 detached HEAD、无法确定分支、目录不是 Git 仓库，或 Git 状态异常时，说明实际状态并停止修改，等待用户决定。
7. 用户确认分支后如果检测到当前分支发生变化，立即停止修改并重新请求确认。
8. 分支确认不等于允许清理工作区。保留所有现有修改；若未提交修改与当前单元涉及同一文件或可能影响实现，先说明冲突或风险，再等待用户决定。

## 控制开发边界

- 只修改当前单元所必需的代码和测试，不重新拆分需求，不调整用户确定的 PR 顺序。
- 不顺带修复、重构或清理范围外问题；发现此类问题时记录并在完成汇报中说明。
- 保护用户已有修改，不覆盖、不回退与当前任务无关的工作区内容。
- 不创建或切换 Git 分支；分支检查和确认遵循“确认开发分支”。
- 不自动提交、推送代码或创建远程 PR，除非用户明确要求。
- 不编写设计文档。实现需要的代码注释、接口说明、迁移或 SQL 文件不属于设计文档。
- 不使用 `superpowers` 或其提供的任何技能，除非用户明确要求使用。
- 遵循仓库已有架构、编码风格、测试方式和项目级指令；项目规则与本技能冲突时，先向用户说明冲突，不自行绕过约束。

## 处理数据库变更

### 确定设计依据

1. 涉及数据库设计或变更时，先查找并遵循项目已有数据库规范，包括仓库指令、技术文档、SQL 目录、迁移文件、ORM 模型和同模块现有表结构。
2. 明文数据库规范优先，不使用通用做法替代项目的明确约定。
3. 没有明文规范时，从当前模块及相邻实现中归纳事实规范，包括：
   - 表名、字段名、主键和业务标识的命名方式；
   - 字段类型、长度、精度、是否允许为空及默认值；
   - 时间、审计和软删除字段；
   - 索引、唯一约束、外键和级联规则的使用方式；
   - 字符集、排序规则、数据库方言和迁移工具；
   - SQL 或迁移文件的目录、命名和编写风格。
4. 现有实现不一致时，选择与当前模块、相邻表和本次业务最接近的模式，不借当前功能统一或重构无关结构，并在完成汇报中说明选择依据。
5. 没有明文规范且没有可参考实现时，采用当前数据库和技术栈的通行做法，同时保持变更最小、向后兼容且便于回滚。不自行添加与需求无关的字段、索引、外键或级联规则。
6. 不凭空假定业务唯一性、空值语义、默认值、精度、码值或删除语义。无法从需求和代码可靠确定且选择会影响业务数据时，先向用户确认。

### 写全中文注释

1. 本次新增或修改的表、字段和码值必须具有完整、准确的中文注释，不得存在未注释项。
2. 表注释应说明业务用途和数据范围；每个字段都必须有中文注释，包括主键、审计字段和软删除字段。
3. 字段注释不得只是字段名的直译。根据实际情况写明业务含义、单位、格式、精度、空值含义、默认值含义、取值范围以及关联标识所指向的业务对象。
4. 枚举、状态、类型等码值必须逐项写明中文含义，例如“状态：0-待处理，1-处理中，2-已完成”。不得只说明字段是“状态”或“类型”。
5. 码值注释必须与代码中的枚举、常量、字典配置和接口定义一致。无法确认含义时询问用户，不根据名称猜测。
6. 使用项目已有数据库方言和迁移方式添加注释。数据库不支持直接注释时，按照项目现有方式在迁移元数据、ORM 描述或配套码表中保存中文说明。
7. 修改现有对象时，保证本次新增或修改的表、字段和码值注释完整。发现范围外的历史注释缺失时只在完成汇报中列出，不擅自扩大修改范围。
8. 生成 SQL 或迁移文件后，逐项检查本次涉及的表、字段和码值，确认没有遗漏、含糊或与实现不一致的注释。

### 控制数据库操作权限

- 可以在当前功能范围内编写 SQL 或迁移文件，但不得自动执行 SQL、应用迁移、回填历史数据或修改数据库数据。
- 执行任何数据库变更前，说明准备执行的内容、影响范围和风险，并获得用户明确授权。
- 删除表或字段、修改字段类型、批量修复数据、调整唯一约束或外键、引入级联删除、建立跨模块数据库依赖等高风险设计，即使只生成方案或文件，也要在无法从已确认需求确定时先询问用户。

## 验证并交付

1. 只运行与当前改动直接相关的聚焦测试，不自动运行全量测试。若没有可运行的聚焦测试，说明原因和未验证风险。
2. 不因测试发现范围外问题而扩大实现；记录问题并交由用户决定。
3. 开发和聚焦测试结束后，根据本次实际改动生成一条简洁、具体的中文 Git 提交语句，固定使用 `feat:` 前缀。只生成语句，不执行提交。
4. 按“识别下一开发单元”查找并报告下一待开发功能点或 PR。
5. 汇报后停止，等待用户确认或指定下一个开发单元。

## 识别下一开发单元

1. 从当前需求文档、功能点列表、PR 路线图、任务清单或用户明确指定的其他来源中查找开发顺序和完成状态。
2. 优先使用项目明确标注的顺序和状态，不根据模型偏好重新排序，不自行新增、合并或拆分开发单元。
3. 以当前开发单元在列表中的位置为基准，跳过明确标记为已完成、已取消或不再开发的项目，将其后第一个待开发项目识别为下一开发单元。
4. 如果下一项目存在前置依赖、阻塞项或尚未满足的条件，同时说明这些信息，不擅自改选其他项目。
5. 不因本次代码已经完成就自动修改需求文档、任务列表或 PR 状态；是否更新状态由用户决定。
6. 找不到功能点列表、无法定位当前单元、存在多个相互冲突的列表或状态不明确时，不猜测下一项。在汇报中写明“无法确定”及具体原因。
7. 只告知下一开发单元，不开始分析或实现它。能够确定时，同时给出下一次简短调用语句，例如 `$dev-flow PR5`。

使用以下结构完成汇报：

```text
实现内容：
聚焦测试：
风险或遗留项：
建议提交语句：feat: ...
下一开发单元：<编号和名称；无法确定时说明原因>
下次调用：$dev-flow <开发单元>
```

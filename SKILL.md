# 开发流程

## 强制工作流

以下规则优先于本 Skill 内其他流程说明：

1. 对来自 `refine-reqs` 或已经引用结构化 SPEC 的开发单元，未找到唯一、已冻结且与当前 PR 匹配的需求基线前，严禁编写或修改业务实现代码。
2. 编码前必须把当前开发单元对应的 `F-xxx`、`C-xxx`、`DC-xxx`、`AC-xxx` 从冻结 SPEC 和 PR 列表中显式拉取并回显，形成《本次开发必须满足的不可协商清单》。
3. 每个适用 `C-xxx` 都必须有实现证据和验证证据。存在缺口时先在当前范围内补齐并重新验证；无法补齐时以阻塞结束，不得生成完成态汇报、提交语句或下一开发单元。
4. 开发过程中出现新增或变更的业务功能、规则、范围或约束时，停止扩展实现，说明冻结基线需要解冻并返回需求澄清流程；不得把新需求静默并入当前 PR。
5. 不为追踪而机械地在源代码中加入 `// 实现 F-xxx` 等注释。每次开始实现时在对话中列出实现依据，交付时通过文件、符号、配置、数据库对象和测试证据建立追踪。

## 核心原则

将本技能的调用视为当前开发单元的编码授权，但不视为对当前 Git 分支或需求基线完整性的确认。默认用户已经指定要开发的功能点或 PR；只有分支确认、冻结需求挂载和范围核对全部通过后才直接开发，不重复规划整个需求，也不设置范围的二次审批。

始终由用户决定开发单元、串行顺序和推进节奏。一次只完成一个用户指定的开发单元，完成后只告知下一待开发单元并停止，等待用户确认，不提前开发后续单元。

使用简短调用格式：`$dev-flow <开发单元>`，例如 `$dev-flow PR4`、`$dev-flow 登录超时修复`。将技能名后的内容直接视为当前开发单元标识，不要求用户补写“使用”“开发”或版本号等重复信息。

## 开始开发

1. 读取适用的仓库规则，并按“确认开发分支”完成分支安全检查。分支确认前只执行只读检查，不修改任何项目文件。
2. 以技能名后的内容为开发单元标识，读取当前仓库的需求记录、SPEC、功能点或 PR 列表、相关代码和测试，并检查当前工作区状态。
3. 优先使用 PR 中记录的冻结基线路径和版本定位 SPEC；只有 PR 未提供引用时才在当前仓库和当前对话中查找。标识、基线或版本无法唯一定位时，不猜测，暂停并请求用户澄清。
4. 根据现有证据简短说明当前单元的目标、范围和明确的非目标，以此完成范围核对。
5. 按“挂载冻结需求”提取并核对当前单元对应的功能、硬性约束、确认决策和验收标准。
6. 比较挂载内容、项目规则和代码现状。代码当前行为与目标约束不同通常是本次修改目标，不单独视为冲突；只有权威需求互相矛盾、冻结基线与项目强制规则冲突、修改超出当前范围或需要额外授权时才暂停。
7. 输出《需求约束挂载》清单给用户过目。清单完整且无冲突时不等待二次确认；在分支已确认的前提下立即编码。
8. 仅在范围缺失、基线未冻结、约束映射缺失、信息互相冲突，或必须由用户作出重要产品或数据决策时暂停并提问。
9. 普通技术细节应遵循项目现有模式自行判断，不反复请求用户决定。

## 挂载冻结需求

编码前必须建立当前开发单元的显式需求挂载清单：

1. 验证需求基线状态为“已冻结”，记录 SPEC 路径或当前对话基线标识、版本以及当前 PR 编号。
2. 从当前 PR 的“对应功能”“适用约束”“对应确认决策”“验收标准”字段读取 `F-xxx`、`C-xxx`、`DC-xxx`、`AC-xxx`，再回到 SPEC 核对编号、原文和适用范围。
3. 对已有需求文档中的 `REQ-xxx` 保持兼容，将其作为功能需求读取；不得擅自改号。
4. 全局 `C-xxx` 必须挂载到所有适用 PR；PR 声明“无适用约束”时，必须能从冻结 SPEC 和映射表证明该结论。

严格区分以下情况：

- 找到唯一冻结 SPEC，且当前 PR 明确声明并能证明没有适用 `C-xxx`：输出“当前开发单元无适用硬性约束”，可以继续；
- 找到 SPEC，但 PR 缺少约束字段、编号不存在或映射不完整：视为需求追踪断裂，停止编码；
- 找不到 SPEC、找到多个候选版本、SPEC 未冻结或 PR 与基线版本不一致：视为基线不明确，停止编码；
- 旧项目确实没有结构化 SPEC 时，从用户明确指定的需求、缺陷记录和验收标准建立“旧版临时挂载清单”，明确标记来源和未结构化风险；涉及新的业务决策时停止并转回需求冻结流程。

使用以下格式回显：

```text
【需求约束挂载】
冻结基线：<SPEC 路径或对话基线标识> @ <版本>（已冻结）
当前开发单元：PR-xx
对应功能：F-xxx / 兼容 REQ-xxx
确认决策：DC-xxx
硬性约束：
- C-001：<完整原文>（来源：SPEC <版本>；适用范围：<范围>）
验收标准：AC-xxx
结论：挂载完整，无冲突，编码时严格遵循。
```

当前代码中的默认值、状态判断或既有行为与目标 `C-xxx` 不同，且修改属于当前 PR 时，将差异列为实现任务并继续。只有无法从当前范围内消解的权威来源冲突、数据风险、权限问题或外部依赖才阻塞。

## 确认开发分支

1. 任何项目文件修改前，使用只读 Git 命令确认当前仓库、当前分支和工作区状态。
2. 将“用户明确确认”作为开始编码的必要条件。满足以下任一情况才视为分支已经确认：
   - 用户在当前请求或当前对话上下文中明确指定了目标分支，且当前分支与其完全一致；
   - 模型报告当前分支名称后，用户明确回复该分支正确或同意在该分支开发。
3. 不根据版本号、PR 名称、分支命名习惯、远程默认分支、历史对话中的其他任务或模型推测，擅自认定当前分支正确。
4. 用户尚未指定或确认目标分支时，报告当前分支名称、工作区是否存在未提交修改，并请求确认。得到确认前停止编码，但可以继续进行不会修改文件的需求和代码检查。
5. 当前分支与用户指定分支不一致时，明确报告两者并停止。不得自行创建、切换、重命名、合并、变基或删除分支；等待用户处理或明确指令。
6. 处于 detached HEAD、无法确定分支、目录不是 Git 仓库，或 Git 状态异常时，说明实际状态并停止修改，等待用户决定。
7. 用户确认分支后如果检测到当前分支发生变化，立即停止修改并重新请求确认。
8. 分支确认不等于允许清理工作区。保留所有现有修改；若未提交修改与当前单元涉及同一文件或可能影响实现，先说明冲突或风险，再等待用户决定。

## 控制开发边界

- 只修改当前单元所必需的代码和测试，不重新拆分需求，不调整用户确定的 PR 顺序。
- 每项新增或修改的业务行为必须能追溯到已挂载的 `F-xxx`、`C-xxx`、兼容 `REQ-xxx` 或完成当前单元所必需的技术工作；无法追溯的行为视为范围外变更。
- 用户在开发中提出新的业务功能、规则或约束时，不直接实现；报告其不在冻结基线内，并等待需求解冻、修订和重新冻结。
- 不顺带修复、重构或清理范围外问题；发现此类问题时记录并在完成汇报中说明。
- 保护用户已有修改，不覆盖、不回退与当前任务无关的工作区内容。
- 不创建或切换 Git 分支；分支检查和确认遵循“确认开发分支”。
- 不自动提交、推送代码或创建远程 PR，除非用户明确要求。
- 不编写设计文档。实现需要的代码注释、接口说明、迁移或 SQL 文件不属于设计文档。
- 不使用 `superpowers` 或其提供的任何技能，除非用户明确要求使用。
- 遵循仓库已有架构、编码风格、测试方式和项目级指令；项目规则与本技能冲突时，先向用户说明冲突，不自行绕过约束。

## 处理数据库变更

### 确定设计依据

1. 涉及数据库设计或变更时，先查找并遵循项目已有数据库规范，包括仓库指令、技术文档、SQL 目录、迁移文件、ORM 模型和同模块现有表结构。
2. 明文数据库规范优先，不使用通用做法替代项目的明确约定。
3. 没有明文规范时，从当前模块及相邻实现中归纳事实规范，包括：
   - 表名、字段名、主键和业务标识的命名方式；
   - 字段类型、长度、精度、是否允许为空及默认值；
   - 时间、审计和软删除字段；
   - 索引、唯一约束、外键和级联规则的使用方式；
   - 字符集、排序规则、数据库方言和迁移工具；
   - SQL 或迁移文件的目录、命名和编写风格。
4. 现有实现不一致时，选择与当前模块、相邻表和本次业务最接近的模式，不借当前功能统一或重构无关结构，并在完成汇报中说明选择依据。
5. 没有明文规范且没有可参考实现时，采用当前数据库和技术栈的通行做法，同时保持变更最小、向后兼容且便于回滚。不自行添加与需求无关的字段、索引、外键或级联规则。
6. 不凭空假定业务唯一性、空值语义、默认值、精度、码值或删除语义。无法从需求和代码可靠确定且选择会影响业务数据时，先向用户确认。

### 写全中文注释

1. 本次新增或修改的表、字段和码值必须具有完整、准确的中文注释，不得存在未注释项。
2. 表注释应说明业务用途和数据范围；每个字段都必须有中文注释，包括主键、审计字段和软删除字段。
3. 字段注释不得只是字段名的直译。根据实际情况写明业务含义、单位、格式、精度、空值含义、默认值含义、取值范围以及关联标识所指向的业务对象。
4. 枚举、状态、类型等码值必须逐项写明中文含义，例如“状态：0-待处理，1-处理中，2-已完成”。不得只说明字段是“状态”或“类型”。
5. 码值注释必须与代码中的枚举、常量、字典配置和接口定义一致。无法确认含义时询问用户，不根据名称猜测。
6. 使用项目已有数据库方言和迁移方式添加注释。数据库不支持直接注释时，按照项目现有方式在迁移元数据、ORM 描述或配套码表中保存中文说明。
7. 修改现有对象时，保证本次新增或修改的表、字段和码值注释完整。发现范围外的历史注释缺失时只在完成汇报中列出，不擅自扩大修改范围。
8. 生成 SQL 或迁移文件后，逐项检查本次涉及的表、字段和码值，确认没有遗漏、含糊或与实现不一致的注释。

### 控制数据库操作权限

- 可以在当前功能范围内编写 SQL 或迁移文件，但不得自动执行 SQL、应用迁移、回填历史数据或修改数据库数据。
- 执行任何数据库变更前，说明准备执行的内容、影响范围和风险，并获得用户明确授权。
- 删除表或字段、修改字段类型、批量修复数据、调整唯一约束或外键、引入级联删除、建立跨模块数据库依赖等高风险设计，即使只生成方案或文件，也要在无法从已确认需求确定时先询问用户。

## 验证并交付

1. 只运行与当前改动直接相关的聚焦测试，不自动运行全量测试。若没有可运行的聚焦测试，说明原因和未验证风险。
2. 聚焦测试结束后，对本次挂载的功能和约束执行正向覆盖检查：每个适用 `F-xxx`、`C-xxx` 或兼容 `REQ-xxx` 都必须同时具有实现证据和验证证据。
3. 实现证据可以是本次新增或修改的代码，也可以是已存在且经核对仍有效的代码、配置、常量、枚举、数据库对象、迁移约束或公共能力；必须记录具体文件路径和符号、键名或对象名，不能只写“已实现”。
4. 验证证据优先使用聚焦测试及具体测试名。数值或阈值约束应验证精确值和边界，状态或权限约束应验证允许与拒绝路径；无法自动测试时说明原因并提供可复现的替代检查。
5. 执行反向范围检查：本次每项新增或改变的业务行为都必须能追溯到挂载的 F/C/REQ 或当前单元必要技术工作，发现无依据实现时移除或报告范围冲突。
6. 发现覆盖缺口时，若属于当前范围，继续补充实现和测试并重新执行聚焦测试与覆盖检查；只有因需求冲突、范围不足、缺少授权或外部条件无法补齐时，才在风险或遗留项中标记 `【阻塞】`。
7. 存在阻塞项时不得宣称当前单元完成，不得生成提交语句或识别下一开发单元。明确报告未覆盖编号、缺失证据和阻塞原因后停止。
8. 不因测试发现范围外问题而扩大实现；记录问题并交由用户决定。
9. 全部覆盖后，根据本次实际改动生成一条简洁、具体的中文 Git 提交语句，固定使用 `feat:` 前缀。只生成语句，不执行提交。
10. 按“识别下一开发单元”查找并报告下一待开发功能点或 PR，然后停止并等待用户确认。

交付时必须展示“开发单元到实现证据”覆盖矩阵：

```markdown
| 需求或约束 | 来源决策 | 开发单元 | 实现证据 | 验证证据 | 状态 |
|---|---|---|---|---|---|
| F-001 | DC-001 | PR-01 | `app/service.py: create_order` | `tests/test_service.py: test_create_order` | 通过 |
| C-001 | DC-002 | PR-01 | `app/config.py: timeout_minutes` | `tests/test_timeout.py: test_closes_at_30_minutes` | 通过 |
```

## 识别下一开发单元

1. 优先从当前冻结基线对应的功能点列表中查找开发顺序和完成状态；没有结构化基线时，再使用需求文档、PR 路线图、任务清单或用户明确指定的其他来源。
2. 优先使用项目明确标注的顺序和状态，不根据模型偏好重新排序，不自行新增、合并或拆分开发单元。
3. 以当前开发单元在列表中的位置为基准，跳过明确标记为已完成、已取消或不再开发的项目，将其后第一个待开发项目识别为下一开发单元。
4. 如果下一项目存在前置依赖、阻塞项或尚未满足的条件，同时说明这些信息，不擅自改选其他项目。
5. 不因本次代码已经完成就自动修改需求文档、任务列表或 PR 状态；是否更新状态由用户决定。
6. 找不到功能点列表、无法定位当前单元、存在多个相互冲突的列表或状态不明确时，不猜测下一项。在汇报中写明“无法确定”及具体原因。
7. 只告知下一开发单元，不开始分析或实现它。能够确定时，同时给出下一次简短调用语句，例如 `$dev-flow PR5`。

使用以下结构完成汇报：

```text
实现内容：
冻结基线：<路径或对话标识 @ 版本>
约束实现：<已实现 C-xxx，覆盖 n/n；或当前单元无适用硬性约束>
需求覆盖矩阵：<逐项列出 F/C/REQ -> 文件:符号 -> 测试/检查 -> 状态>
反向范围检查：<全部改动有需求依据 / 列出异常>
聚焦测试：
风险或遗留项：
建议提交语句：feat: ...
下一开发单元：<编号和名称；无法确定时说明原因>
下次调用：$dev-flow <开发单元>
```

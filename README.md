# dev-flow

`dev-flow` 是一个面向用户主导、串行开发场景的 Codex Skill。它读取 `refine-reqs` 形成的冻结 SPEC 和功能点 PR，在编码前挂载当前单元的功能、硬性约束、确认决策和验收标准，在交付前生成需求到代码与测试证据的覆盖矩阵。

## 主要约束

- 开发前必须确认当前 Git 分支，未经确认不修改文件。
- 来自结构化需求流程的开发单元必须能定位唯一、已冻结且版本匹配的 SPEC。
- 编码前必须挂载当前 PR 对应的 `F-xxx`、`C-xxx`、`DC-xxx` 和 `AC-xxx`。
- 找不到 SPEC、版本冲突或约束映射缺失时停止编码；不能把“没找到约束”解释为“没有约束”。
- 一次只开发一个功能点或 PR，不提前处理后续单元。
- 开发中出现新业务需求时停止扩展范围，先解冻并修订需求基线。
- 不创建或切换分支，不自动提交、推送或创建远程 PR。
- 不编写设计文档，不使用 `superpowers`。
- 数据库设计遵循项目规范，表、字段和码值必须具有完整中文注释。
- 可以生成 SQL 或迁移文件，但执行数据库变更前必须获得用户授权。
- 完成后只运行聚焦测试，并为每项需求和约束提供实现与验证证据；存在未覆盖项时不得生成完成态汇报或提交语句。

## 默认流程

```text
确认 Git 分支
-> 定位当前 PR 和冻结 SPEC
-> 核对目标、范围和 Non-goals
-> 挂载 F/C/DC/AC
-> 无冲突时直接编码
-> 运行聚焦测试
-> 执行正向覆盖与反向范围检查
-> 输出需求覆盖矩阵
-> 全部通过后生成提交语句并提示下一单元
```

## 需求约束挂载

编码前会先输出：

```text
【需求约束挂载】
冻结基线：docs/order-spec.md @ v1.0（已冻结）
当前开发单元：PR-01
对应功能：F-001
确认决策：DC-001
硬性约束：
- C-001：工单超时关闭阈值为30分钟
验收标准：AC-001
结论：挂载完整，无冲突，编码时严格遵循。
```

当前代码仍使用15分钟而冻结 SPEC 要求30分钟时，这通常是本次实现目标，不自动视为需求冲突。只有权威需求互相矛盾、修改超出当前 PR、违反仓库强制规则或缺少必要授权时才阻塞。

## 三段追踪映射

### 决策到需求

该映射来自冻结 SPEC：

| 确认决策 | 用户结论 | SPEC 落点 |
|---|---|---|
| `DC-001` | 超时关闭阈值为30分钟 | `C-001` |
| `DC-002` | 支持待处理工单撤回 | `F-002`、`C-002` |

### 需求到开发单元

该映射来自功能点 PR 列表：

| 功能开发点 | 对应功能 | 适用约束 | 确认决策 | 验收标准 |
|---|---|---|---|---|
| `PR-01` 超时处理 | `F-001` | `C-001` | `DC-001` | `AC-001` |
| `PR-02` 工单撤回 | `F-002` | `C-002` | `DC-002` | `AC-002` |

### 开发单元到实现证据

每个当前 PR 完成后必须输出：

| 需求或约束 | 来源决策 | 开发单元 | 实现证据 | 验证证据 | 状态 |
|---|---|---|---|---|---|
| `F-001` | `DC-001` | `PR-01` | `app/service.py: process_timeout` | `tests/test_timeout.py: test_timeout_close` | 通过 |
| `C-001` | `DC-001` | `PR-01` | `app/config.py: timeout_minutes` | `tests/test_timeout.py: test_closes_at_30_minutes` | 通过 |

实现证据可以是代码、配置、常量、枚举、数据库对象或已经存在的公共能力；验证证据优先使用具体测试。数值约束要验证精确值和边界，状态或权限约束要验证允许与拒绝路径。

完整追踪链为：

```text
DC-xxx
-> F-xxx / C-xxx / AC-xxx
-> PR-xx
-> 文件:符号 / 配置 / 数据库对象
-> 测试或其他验证证据
```

同时执行反向检查：每项新增或改变的业务行为都必须能回溯到 F/C、兼容的 `REQ-xxx` 或当前单元必要技术工作，避免范围外实现。

## 缺口处理

- 当前范围内发现某个 F/C 缺少实现或测试时，继续补齐并重新验证，不把自身遗漏直接交给用户。
- 因需求冲突、范围不足、缺少授权或外部条件无法补齐时，以 `【阻塞】` 报告，且不生成提交语句或下一开发单元。
- 冻结 SPEC 明确证明当前 PR 没有适用 C 时，可以输出“当前开发单元无适用硬性约束”。
- 找不到 SPEC 或 PR 缺少映射时不得使用上述结论。

## 安装

将仓库克隆到 Codex 的个人 Skill 目录：

```bash
mkdir -p "${CODEX_HOME:-$HOME/.codex}/skills"
git clone https://github.com/<你的用户名>/dev-flow.git "${CODEX_HOME:-$HOME/.codex}/skills/dev-flow"
```

将示例地址替换为实际 GitHub 仓库地址。安装后重新开启 Codex 会话。

## 使用

```text
$dev-flow PR4
$dev-flow 登录超时修复
```

版本号不是调用参数的必填项；Skill 会优先读取 PR 中记录的冻结基线版本。只有当前上下文无法唯一定位开发单元、SPEC 或版本时，Codex 才会请求补充信息。
